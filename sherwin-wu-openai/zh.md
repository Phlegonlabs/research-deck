# Sherwin Wu — OpenAI API 工程负责人与"巫师"论

## 1. 谁是 Sherwin Wu

### 背景

Sherwin Wu 是 OpenAI API 和开发者平台的工程负责人（Head of Engineering）——这是支撑所有基于 OpenAI 模型构建的第三方应用的基础设施层。他是 OpenAI 的早期成员，从技术人员（Member of Technical Staff）晋升为整个 API 工程组织的负责人。

**教育背景：** MIT 计算机科学学士和工程硕士（2010-2014）。

**职业路径：**

| 时间 | 公司 | 角色 | 核心工作 |
|------|------|------|----------|
| ~2014 | Palantir Technologies | 工程师 | 早期职业（实习/短期） |
| ~2014 | Doximity | 工程师 | 医疗科技 |
| 2014-2016 | Quora | 软件工程师 | 信息流排序与产品团队 |
| 2016-2022 | Opendoor | 产品负责人 → 工程经理 | 定价团队 5 年以上——住宅房地产 ML 定价模型。"一次错误的预测可能损失数百万美元。" |
| ~2022-至今 | OpenAI | 技术人员 → API 平台工程负责人 | 开发者平台团队，从 IC 成长为组织领导者 |

**副业角色：**
- **Sancus Ventures** — 出现在团队页面上（可能是顾问/LP 角色）
- 在 X/Twitter（@sherwinwu）上活跃分享 OpenAI 开发者洞察

### 为什么他重要

Wu 站在 OpenAI 研究与开发者生态的交汇点。他的团队构建了 API、Responses API、Agents SDK、AgentKit 和 1500 万以上开发者使用的开发工具。他拥有独特的观察视角：他同时看到开发者在构建什么、哪些模式有效、以及模型的发展方向。Martin Casado（a16z GP）称他为"AI 领域最优秀的系统思考者之一"。

### 2.3 QCon 纽约 2023 — "AI 思维的自行车：GPT-4 + 工具"（2023 年 6 月）

**来源：** [InfoQ 演讲](https://www.infoq.com/presentations/bicycle-ai-gpt-4-tools/) | [QCon 演讲者页面](https://qconnewyork.com/speakers/sherwinwu)
**日期：** 2023 年 6 月 14 日
**联合演讲者：** Atty Eleti（OpenAI 工程师，前 Stripe）

**核心论点：** 如果乔布斯将计算机称为"思维的自行车"，那么函数调用就是"AI 思维的自行车"——它将 LLM 从孤立的知识盒子扩展为能与现实世界交互的工具。

**关键技术洞察：**

1. **识别的 LLM 局限性：**
   - 训练数据是固定的且过时的
   - 无法访问实时信息
   - 概率性输出使 API 集成不可靠（结构化 JSON 准确率 70-80%）
   - 模型会产生幻觉，有时会在 JSON 前添加解释性文本，导致解析器崩溃

2. **函数调用作为解决方案：**
   - 三步循环：用户查询 → 模型构建函数调用 → 开发者执行 → 模型总结结果
   - 在"数十万"个示例上进行微调以提高可靠性
   - GPT-4 在专业考试中达到第 80-90 百分位"尽管只是预测下一个 token"

3. **三个演示：**
   - **自然语言转 SQL** — 将业务问题转换为可执行查询
   - **多个外部 API** — 组合位置服务 + Yelp API 提供餐厅推荐
   - **代码审查自动化** — GPT 推理应用于专业代码审查，具有可自定义的个性

4. **承认的局限性：**
   - 即使有函数调用，模型仍然容易产生幻觉
   - 没有原生并行函数调用（解决方案：包装函数）
   - 即使使用英语提示，也可能出现非英语输出

**要点：** 这次演讲是函数调用的公开首秀——这个原语后来演变为工具使用，然后是 Structured Outputs，再然后是 Agents SDK。Wu 在根基就在那里。

### 2.7 BG2 Pod — "深入 OpenAI 企业版"（2025 年 9 月）

**来源：** [Spotify](https://open.spotify.com/episode/0HnYWQc9retjyFVSEAmFmQ) | [Apple Podcasts](https://podcasts.apple.com/us/podcast/inside-openai-enterprise-forward-deployed-engineering/id1727278168?i=1000726402888) | [Chain of Thought 总结](https://podcasts.chainofthought.xyz/podcast-summaries/inside-openai-enterprise-forward-deployed-engineering-gpt-5-and-more-bg2-guest-interview)
**日期：** 2025 年 9 月 11 日（约 1 小时 9 分钟）
**联合嘉宾：** Olivier Godement（OpenAI 平台产品负责人）
**主持：** Apoorv Agrawal（Altimeter）

**关键洞察：**

**1. OpenAI 的第一个产品是 API，不是 ChatGPT**
API/平台是分发 AI 效益的主要载体。ChatGPT 是后来的消费者产品。

**2. 前置部署工程（FDE）模式：**
OpenAI 采用了 Palantir 的 FDE 模式——工程师深度嵌入企业客户，构建定制解决方案。
> "非常深入地嵌入客户，构建针对其系统的特定方案。"

FDE 处理系统设计、API 集成，并将模型连接到企业工具——其中许多缺乏干净的 API。

**3. 企业案例：**

| 客户 | 用例 | 关键细节 |
|------|------|----------|
| **T-Mobile** | 自动化文字+语音客服 | 实时 API 实现自然对话；改进反馈到 GPT-5 模型增强 |
| **Amgen** | 加速药物开发+监管文档 | 自动化延迟药物发布的行政繁重流程 |
| **Los Alamos** | 气隙超级计算机上的国家安全研究 | o3 推理模型在 Venado 超级计算机上的定制本地部署 |

关于 Los Alamos 部署：
> "我们实际上不得不将模型的权重物理地带入他们的超级计算机。"

**4. GPT-5 开发哲学：**
> "对于 GPT-5，同样重要和有影响力的是工艺：模型的风格、语气、行为。"

GPT-5 的指令遵循能力太强，产生了新问题：
> "当你把这个给 GPT-5 时，它会想'天哪，这个人真的想简洁'。然后回复只有一句话。"

**5. 强化微调（RFT）：**
RFT 优于传统的有监督微调。它使组织能够利用专有数据开发特定领域的领先模型。

**6. 做多/做空：**
- **做多：** 电竞（亚洲增长潜力，年轻人群文化相关性）
- **做空：** AI 工具创业公司（eval 平台、向量数据库——竞争太激烈且技术快速演变）

**要点：** FDE 模式是 OpenAI 的企业护城河。他们不只是卖 API——他们嵌入工程师，这些工程师学习客户问题并将洞察反馈到模型开发中。T-Mobile → GPT-5 的改进循环是最清晰的例子。

### 2.10 a16z Show — "OpenAI 如何为 8 亿周活用户构建"（2025 年 11 月）

**来源：** [Spotify](https://open.spotify.com/episode/36DXaD9KdlWh2Ksn7OVAdc) | [Apple Podcasts](https://podcasts.apple.com/hn/podcast/how-openai-builds-for-800-million-weekly-users-model/id842818711?i=1000738780411) | [Podwise](https://podwise.ai/dashboard/episodes/6063995) | [01Cloud 分析](https://engineering.01cloud.com/2025/12/22/how-openai-scales-to-800-million-weekly-active-users-insights-from-sherwin-wu-on-model-specialization-and-fine-tuning/)
**日期：** 2025 年 11 月 28 日（约 53 分钟）
**主持：** Martin Casado（a16z GP）

Martin Casado 称 Wu 为"AI 领域最优秀的系统思考者之一"。

**关键洞察：**

**1. "上帝模型"的消亡：**
> "即使在 OpenAI 内部，想法也是会有一个统治一切的模型。这绝对已经完全改变了。越来越清楚的是，将会有空间容纳一系列专门化的模型。"

**2. 模型粘性是真实的：**
> "用户已经知道并关心他们使用的是哪个模型。"

模型是"反去中介化技术"——它们难以被抽象掉，因为用户会与特定模型建立关系。

**3. 强化微调是重大突破：**
> "最近发生的重大突破是强化微调。"

公司有"坐在那里的巨大数据宝库"。RFT 让他们能利用这些数据创建特定领域的模型。

**4. 按用量定价：**
按用量定价仍然是最优的，因为它与实际消费模式一致。成本加成定价确保合理利润。

**5. 平台 vs 产品的张力：**
> "我们希望 ChatGPT 作为第一方应用，这是获得 8 亿周活或现在更多的好方式。全球十分之一的人口，对吧？"

OpenAI 明确管理其消费产品（ChatGPT）和平台（API）之间的张力。

**6. 确定性 Agent 架构：**
OpenAI 倾向于"确定性的、基于节点的" agent 工作流，而非自由漫游系统——特别是对于需要可解释性的受监管行业。

**7. 从提示工程到上下文设计：**
Wu 标志着从"提示工程"到"上下文设计"的根本转变——构建数据和上下文以引导模型表现，而不仅仅是制作巧妙的提示。

**要点：** 这次访谈是了解 OpenAI 平台战略最全面的窗口。从"一个模型"到"模型组合"的转变反映了更广泛的行业趋势。RFT 洞察解释了为什么 OpenAI 在微调 API 上大量投资。

### 2.14 Lenny's Podcast — "工程师正在成为巫师"（2026 年 2 月）

**来源：** [Lenny's Newsletter](https://www.lennysnewsletter.com/p/engineers-are-becoming-sorcerers) | [Spotify](https://open.spotify.com/episode/3EPDh5GqC9PEdyQBEpmdrQ) | [Apple Podcasts](https://podcasts.apple.com/us/podcast/engineers-are-becoming-sorcerers-the-future-of/id1627920305?i=1000749436380) | [Podwise](https://podwise.ai/dashboard/episodes/7171472)
**日期：** 2026 年 2 月
**主持：** Lenny Rachitsky

这是 Wu 最全面、传播最广的访谈。标题本身——"工程师正在成为巫师"——已成为 AI 工程社区的 meme。

**关键数据：**

| 指标 | 数值 |
|------|------|
| 每天使用 Codex 的 OpenAI 工程师 | 95% |
| 每个工程师的并行 AI agent 数量 | 10-20 个 |
| 由 Codex 审查的 PR | 100% |
| Codex 用户比非用户多开的 PR | 70% |
| 代码审查时间缩短 | 10-15 分钟 → 2-3 分钟 |

**核心论点——"工程师是巫师"：**
> "工程师正在成为技术负责人。他们管理着一支又一支 agent 大军。"

这个比喻引用了《魔法师的学徒》：
> "这真的感觉像我们是施法的巫师，这些法术就像是出去替你做事。"

**一个人的十亿美元创业公司：**
Wu 对这个叙事持悲观态度。相反，他预测了创业扩散效应：
> "如果一个人创建一个十亿美元创业公司变得容易了，那也意味着人们创建一般的创业公司变得更加容易。"

> "将会出现一个巨大的创业潮和小型 SMB 风格的繁荣，任何人都可以为任何事情构建软件。"

> "可能会有一百个百万美元的创业公司。可能会有成千上万个一千万美元的创业公司……拥有一个一千万美元的业务其实相当不错。"

> "我认为我们可能真的会进入 B2B SaaS 的黄金时代。"

**"模型会把你的脚手架当早餐吃掉"：**
引用 Nicolas Bustamante 的概念——开发者今天围绕模型构建的脚手架/包装器将被模型本身在改进时吸收。

**为模型的未来方向构建：**
> "确保你是为模型的发展方向构建，而不是为它们今天的状态构建。"

> "这是模型有史以来最差的时候。"（引用 Kevin Whale）

**管理转型：**
- 将 50% 以上的时间花在顶尖人才上，像"外科医生"一样赋能他们（《人月神话》参考）
- AI 采用需要**自上而下的买入 + 自下而上的热情**兼具
- 建立 AI 传道者的内部"老虎团队"
- 避免没有草根热情的自上而下强制

**100% AI 编写代码库的挑战：**
维护完全由 Codex 编写的代码库的团队在没有手动回退的情况下面临独特挑战。解决方案：
> "添加文档并将部落知识编码到代码库中。"

**未来 18 个月预测：**
- Agent 运行时间更长，任务连贯性增加
- 多模态音频的重大改进
- 模型处理数小时任务的连贯性

## 3. 所有内容中的关键主题

### 主题 1：工程师作为编排者，而非编码者

Wu 的核心论点在三年中不断演进：
- **2023（QCon）：** 函数调用扩展 AI 能力——LLM 获得"工具"
- **2024（o1 推文）：** 推理模型改变产品设计范式
- **2025（BG2/a16z）：** 企业采用需要 FDE + 模型定制
- **2026（Lenny's）：** 工程师是管理 10-20 个 agent 大军的"巫师"

进程：**工具 → 推理 → 定制 → 编排**。每一层都建立在前一层之上。到 2026 年，工程师的工作不是写代码，而是编排 AI 系统。

### 主题 2：模型组合优于上帝模型

Wu 一直在远离"一个模型统治一切"的论点：
- 特定任务的专门化模型
- 微调（特别是 RFT）作为专门化的机制
- 用户关心他们使用的是哪个模型——模型是"反去中介化技术"

### 主题 3：为模型的未来方向构建

Wu 给开发者最可行动的建议：
- "这是模型有史以来最差的时候"
- "模型会把你的脚手架当早餐吃掉"
- 不要过度投资于当前模型限制的变通方案
- 你今天构建的脚手架将被明天的模型吸收

### 主题 4：前置部署工程作为护城河

OpenAI 的企业战略深受 Palantir FDE 模式的启发：
- 工程师嵌入客户
- 客户问题指导模型改进（T-Mobile → GPT-5）
- 部署和研究之间的反馈循环才是真正的护城河

### 主题 5：开发者作为 AGI 分发层

Wu 围绕一个理念构建了 OpenAI 的整个平台战略：
- ChatGPT 触达消费者（8 亿周活）
- 但 API 通过开发者触达*每个垂直领域*
- OpenAI 无法为每个行业构建——开发者将 AGI 带入利基市场
- 这就是为什么平台投资与模型同样重要

### 主题 6：从提示工程到上下文设计

开发者技能的演进：
- **2023：** 提示工程（制作巧妙的提示）
- **2024：** 函数调用/工具使用（连接外部世界）
- **2025：** 上下文工程（为模型消费构建数据）
- **2026：** Agent 编排（管理 AI agent 大军）

### 主题 7：小型创业公司的黄金时代

Wu 关于"一个人的十亿美元公司"的逆向思维：
- 他对独立创始人独角兽持悲观态度
- 对数千个 1000 万-1 亿美元的利基业务持乐观态度
- AI 降低了为任何垂直领域构建软件的门槛
- "B2B SaaS 的黄金时代"

## 4. 可行动的要点

### 面向开发者

1. **现在就开始管理 agent。** OpenAI 自己的工程师运行 10-20 个并行 Codex agent。如果你还在逐行写代码，你已经落后了。"巫师"模式——施放法术（agent 任务）并监控结果——是新的工作流程。

2. **不要过度投资脚手架。** 你今天构建的任何包装器、RAG 管道或提示链，模型可能在 12-18 个月内吸收它。构建最简单的可工作版本。"模型会把你的脚手架当早餐吃掉。"

3. **学习上下文设计，不仅仅是提示工程。** 转变是从"如何写巧妙的提示"到"如何构建数据使模型产生正确输出"。这包括系统提示、检索上下文、工具描述和输出 schema。

4. **用最新模型测试。** GPT-5 的指令遵循太强，以至于旧的提示技巧（恳求、威胁、角色扮演）现在会适得其反。每一代模型都需要提示词重新校准。

5. **在代码库中编码部落知识。** 随着 AI 编写更多代码，最大的风险是机构知识的丢失。Wu 的团队通过直接在代码库中嵌入文档和上下文来解决这个问题。

### 面向创始人

1. **走垂直，走利基。** Wu 认为 AI 使"多一百倍的创业公司"成为可能——不是更大的公司，而是更专业化的公司。获胜策略是深度垂直专长 + AI 杠杆。

2. **1000 万美元的业务是新的十亿美元愿景。** Wu 明确说"拥有一个一千万美元的业务其实相当不错。" VC 规模的独角兽不是唯一的路径。

3. **采用自上而下 + 自下而上。** 仅靠强制命令的 AI 转型会失败。建立热情者的"老虎团队"，给他们资源，让采用自然扩散。

4. **不要构建 AI 工具创业公司。** Wu 明确做空"eval 平台、向量数据库"——竞争格局变化太快，OpenAI 会吸收这些原语。

### 面向工程领导者

1. **将 50% 以上的时间花在顶尖人才上。** Wu 的管理哲学：像"外科医生"一样对待顶尖人才（《人月神话》）。给他们杠杆，而不是流程。

2. **诚实衡量 AI 采用。** OpenAI 的 95% Codex 采用率、100% AI PR 审查、70% 更多 PR。这些是真实指标——为你的组织建立类似的基准。

3. **准备好应对"太好了"问题。** 随着模型改进，挑战从"让模型做你想要的"变为"模型完全按照你说的做，字面意思上"。这需要更清晰、更精确的沟通——不是更有创意的提示。

4. **先用 Codex 做代码审查。** Wu 的团队将审查时间从 10-15 分钟缩短到 2-3 分钟。代码审查是 AI 采用风险最低、投资回报率最高的起点。
