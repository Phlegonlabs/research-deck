# Jito：Solana 的 MEV 基础设施与流动性质押协议 — 深度分析

## 1. Jito 是什么 — 产品深度解析

Jito 是 Solana 最主导的 MEV（最大可提取价值）基础设施栈，同时也是网络上最大的流动性质押协议。截至 2026 年初，超过 95% 的 Solana 质押权重运行 Jito-Solana 验证者客户端，使其成为 Solana 上最关键的第三方基础设施。

### 1.1 Jito-Solana 验证者客户端

基于 Agave 验证者客户端（原 Solana Labs 客户端）的 Rust 分叉，增加了三个额外的管道阶段：

| 阶段 | 用途 |
|------|------|
| **RelayerStage** | 通过 gRPC 连接 Jito Relayer 进行交易摄入 |
| **BlockEngineStage** | 从区块引擎拍卖接收获胜 bundle |
| **BundleStage** | 在区块生产管道中原子性处理 bundle |

性能与原版 Agave 完全一致 — 增加的部分纯粹与 MEV 相关。客户端于 2022 年 10 月 27 日开源，经 Neodyme 和 Halborn 审计。

**时间线：**
- 2022 Q2：与选定验证者进行私有测试
- 2022 年 10 月：开源
- 2023 年：快速采用，突破 50% 质押份额
- 2024 年：90%+ 质押份额
- 2025-2026 年：95-97% 质押权重

### 1.2 Jito 区块引擎（Block Engine）

区块引擎是一个**链下区块空间拍卖** — 核心 MEV 协调机制。

```
                            ┌─────────────┐
                            │  搜索者      │
                            │ (提交       │
                            │  bundle +   │
                            │  小费)      │
                            └──────┬───────┘
                                   │
                            ┌──────▼───────┐
  ┌──────────┐              │   区块引擎    │
  │  用户    │──────────►   │              │
  │(常规    │   Relayer     │ 1. 验证      │
  │ 交易)   │──(200ms)──►  │ 2. 模拟      │
  └──────────┘              │ 3. 拍卖      │
                            │ 4. 选择      │
                            └──────┬───────┘
                                   │
                            ┌──────▼───────┐
                            │   验证者      │
                            │ (Jito-Solana │
                            │  客户端)     │
                            └──────────────┘
```

**拍卖过程：**
1. **基本检查** — 验证交易，确保 bundle 包含 5 笔或更少交易
2. **模拟** — 验证所有交易成功，计算小费金额
3. **拍卖** — 按状态锁分组 bundle，模拟组合，选择出价最高的 Top-N bundle
4. **交付** — 将获胜 bundle 流式传输到验证者的 BundleStage

区块引擎是**部分区块**拍卖 — 不像以太坊的 MEV-Boost 构建完整区块，Jito 在一个区块内为 bundle 包含进行粒度化拍卖。一个区块中可以发生多次拍卖。

### 1.3 Jito Bundles

Bundle = 最多 5 笔交易，**原子性且顺序执行**：

- **全部成功或全部失败**：任何交易失败，整个 bundle 被丢弃
- **有序**：交易按指定的精确顺序执行
- **需要小费**：最后一笔交易必须包含给验证者的小费

常见 bundle 模式：
- **套利**：在 DEX A 买入 + 在 DEX B 卖出（2 笔交易）
- **清算**：观察欠抵押仓位 + 清算（2 笔交易）
- **三明治攻击**：前跑 + 受害者 + 后跑（3 笔交易，现在是 BAM 的打击目标）

### 1.4 Jito Relayer

Relayer 是一个 **TPU（交易处理单元）代理**，位于用户和验证者之间：

- 作为独立服务运行，通过 gRPC 连接验证者
- 执行去重、过滤和签名验证
- **持有交易约 200ms** 以给区块引擎时间运行 bundle 拍卖
- 每 500ms 发送心跳；如果 ~1.5s 内没有心跳，验证者回退到直接 QUIC
- 使用 Tokio 异步运行时进行非阻塞事件驱动处理
- 负载均衡以防止区块引擎瓶颈

### 1.5 Jito 小费 — 分配机制

两个链上程序处理小费流：

**小费支付程序（Tip Payment Program）：**
- 作为每个验证者 MEV 小费的链上"存钱罐"
- 搜索者将小费转到静态公钥，确保支付给正确的 leader
- 小费以 SOL 计价

**小费分配程序（Tip Distribution Program）：**
- 每个 epoch（约 2-3 天），验证者创建唯一的 `TipDistributionAccount`，由 epoch + 投票账户派生
- epoch 结束后，验证者快照最后一个 slot，生成验证者和质押账户的 Merkle 树
- 将 Merkle 根上传到链上
- 验证者和质押者通过 Merkle 证明领取 MEV 小费（无需许可的领取）

**费用分配（JIP-24 之前）：** 94% 给验证者/质押者，3% 给 Jito Labs，3% 给 Jito DAO
**费用分配（JIP-24 之后，2025 年 8 月）：** 94% 给验证者/质押者，6% 给 Jito DAO 金库

### 1.6 JitoSOL — 流动性质押代币

JitoSOL 是 Solana 最大的 LST — 第一个将 MEV 收益纳入质押收益的代币。

**机制：**
- 用户存入 SOL，收到 JitoSOL
- JitoSOL 数量保持不变；与 SOL 的兑换率随时间增加
- 收益 = 基础质押奖励（约 6-6.6% APY）+ MEV 小费（约 1.2-1.8% APY）= **约 7.2-8.4% 总 APY**
- 协议费：所有奖励的 4%（由 DAO 管理）

**DeFi 集成：**
- 在 Kamino、Drift、Marginfi、Solend 上作为抵押品
- 在 Orca、Raydium 上的 LP 交易对
- 通过 Kamino 的杠杆收益策略
- 借贷平台持有超过 $4.2 亿的 LST = 其总 TVL 的 40.3%

### 1.7 Jito StakeNet

StakeNet = 去中心化的链上质押池管理。替代链下委托机器人。

**评分系统：**
- 4 层分级评分：收益率、佣金率、正常运行时间、MEV 表现
- 每 10 个 epoch 周期选择前 200 名验证者
- 每个选中的验证者获得池 TVL 的 0.5%
- 评分较低的验证者优先被取消质押

**安全机制：**
- 如果投票低于预期 slot 的 70%（不活跃），立即移除
- 如果佣金超过 5% 或 MEV 佣金超过 10%，立即移除
- 无需许可的 cranking — 任何人都可以推进状态机

### 1.8 Jito (Re)staking

面向节点共识网络（NCN）的多资产质押协议 — Solana 对 EigenLayer 的回应。

**三个参与者：**

| 角色 | 功能 |
|------|------|
| **NCN** | 节点运营者集合 + 链上程序，对链下工作负载达成共识 |
| **Vault** | 持有质押的 SPL 代币，委托给 Operator，铸造流动性收据代币（VRT） |
| **Operator** | 执行 NCN 定义的任意逻辑（数据处理、共识、验证） |

**关键特性：**
- 多资产：接受 SOL、JitoSOL、JTO 和其他 SPL 代币
- Vault 产出 Vault Receipt Token（VRT）用于 DeFi 可组合性
- Operator 可同时服务多个 NCN
- NCN 的罚没条件由 Solana 运行时强制执行

**TVL（2025 年 2 月）：** 9 个 VRT 共 $2.27 亿 — ezSOL ($8,620 万), fragSOL ($7,750 万), kySOL ($4,080 万), fragJTO ($1,080 万), kyJTO ($670 万), ezJTO ($500 万)

### 1.9 TipRouter NCN

第一个上线的 NCN — 去中心化的 MEV 小费分配。

**上线日期：** 2025 年 1 月 30 日

**工作原理：**
1. 每个 epoch，NCN 节点运营者独立计算小费分配的 Merkle 根
2. 达到 2/3 共识后，根被提交到链上
3. 分配自动包含 3% 的 DAO/NCN 费用
4. 所有 Jito 小费和区块奖励的 2.7% 流向 Jito DAO
5. 0.30% 路由给 NCN 参与者（在 JTO 质押者和 JitoSOL 再质押者之间分配）

**重要性：** 此前，小费分配是中心化的 — Jito Labs 计算并上传 Merkle 根。TipRouter 将此过程去中心化为独立运营者之间的共识过程。

## 2. Solana 上的 MEV — 技术深度解析

### 2.1 Solana vs 以太坊 MEV 架构

| 维度 | 以太坊 | Solana |
|------|--------|--------|
| **出块时间** | 12 秒 | 400ms slot |
| **内存池** | 公开（所有人可见） | 无（Gulf Stream 直接转发给 leader） |
| **区块构建** | 完整区块拍卖（MEV-Boost） | 部分区块 bundle 拍卖（Jito） |
| **竞争维度** | Gas 价格竞标 | 延迟（先到达 leader） |
| **MEV 面** | 离散（按区块） | 连续（流式） |
| **状态模型** | 全局锁（串行 EVM） | 显式账户锁（并行 Sealevel） |
| **Leader 调度** | 选中前未知 | 提前已知（确定性） |

### 2.2 为什么 Solana 创造了不同的 MEV 动态

**Gulf Stream** — 交易在 slot 开始前直接转发给下一个 leader。没有公共内存池意味着没有公开的交易可见性，理论上限制了前跑。但是...

**Leader 调度** — Solana 的 leader 调度是确定性的，提前 2 个 epoch 已知。搜索者确切知道哪个验证者将产生下一个区块，可以：
- 在已知 leader 附近部署基础设施
- 构建到 leader 的直接网络路径
- 获得以微秒计量的延迟优势

**连续区块构建** — Solana 的 400ms slot 时间加上连续状态更新意味着：
- 快速读取状态更新至关重要
- 机会在毫秒内出现和消失
- **延迟是主要竞争维度**，而非 gas 价格
- 这创造了中心化压力 — 网络连接最快的人获胜

**Sealevel 并行性** — 交易预先声明其账户锁，启用并行执行。MEV 影响：
- Bundle 仅在触及重叠账户时冲突
- 多个不冲突的 bundle 可在同一区块中执行
- 区块引擎可基于账户锁分析优化 bundle 放置

### 2.3 Solana 上的 MEV 类型

**套利（主导）：**
- 跨 DEX 价格差异（Orca、Raydium、Jupiter 池）
- CEX-DEX 套利
- 2024 年，Jito 检测到 9,040 万笔成功套利交易，每笔平均利润 $1.58
- 主要是后跑策略

**清算：**
- 扫描借贷协议（Kamino、Drift、Marginfi）寻找欠抵押仓位
- 原子性清算 + 抵押品获取
- 后跑策略

**三明治攻击：**
- 前跑 + 受害者 + 后跑原子性捆绑
- 利用大额交换的已知价格影响
- 内存池关闭后 16 个月提取了 $3.7-5 亿
- 93% 的三明治攻击是"宽三明治"（跨 slot），每年提取 529K+ SOL

### 2.4 内存池争议（2024 年 3 月）

**时间线：**
- **2022 年 11 月**：Jito 随区块引擎一起启动公共内存池
- **2023-2024 年**：内存池启用三明治攻击；Meme 币交易放大了问题
- **2024 年 3 月 8 日**：MEV 小费单日达 10,000 SOL（$150 万）— 历史最高
- **2024 年 3 月 9 日**：Jito Labs 暂停内存池，声明伤害了零售用户
- **2024 年 6 月**：Solana 基金会从委托计划中移除 32 个三明治攻击验证者（150 万 SOL，占计划质押的 0.5%）
- **2024 年 6 月**：Jito DAO 治理提案将恶意验证者从 JitoSOL 质押池中列入黑名单

**之后发生了什么：**
- 问题转入地下 — 私有内存池出现
- 据报道运营者从私有三明治基础设施中赚取"数十万美元"
- Solana 基金会的委托黑名单被批评为中心化治理
- sandwich.me 等工具发布以跟踪和揭露 MEV 活动

**为什么无法工程化解决：**
- Solana 的架构将交易发送给 leader，而非内存池
- 但任何在 leader 之前看到交易的中介（如区块引擎/Relayer）都创造了事实上的内存池
- 200ms 的 relayer 延迟就是攻击窗口
- 工程化解决需要从根本上重新思考区块构建 — 这就是 BAM

### 2.5 当前 MEV 格局（内存池关闭后）

**收入数据：**
- 2024 年全年：Jito 小费占 Solana 真实经济价值（REV）的约 50%
- 向质押者和验证者支付了 $6.74 亿+
- 2024 年 11 月 17 日：单日峰值 $1,470 万
- 2025 年 1 月 20-21 日：每日费用超过 $2,000 万
- 2025 年 1 月 20 日：单日 MEV 收入峰值 100,000 SOL
- 2024 年 10 月后平均每日 MEV：约 30,000 SOL（最低 10,000 SOL）
- 截至 2025 年 1 月，Jito 占 Solana 费用市场的 60%

```
搜索者                区块引擎              验证者

   │── 提交 Bundle ─────►│                    │

   │                     │── 比较小费         │

   │                     │                    │── BundleStage
   │                     │                    │── 原子执行
   │                     │                    │── 全成功或
   │                     │                    │   全回滚
   │                     │                    │
   │◄── 结果 ────────────┼────────────────────│
```

### 3.3 Jito-Solana 与原版 Agave 的差异

| 组件 | 原版 Agave | Jito-Solana |
|------|-----------|-------------|
| TPU | 直接 QUIC 连接 | Relayer 代理（gRPC） |
| 区块构建 | FIFO 调度 | Bundle 感知调度 |
| 管道阶段 | 标准 4 个阶段 | 标准 4 个 + 3 个 Jito 阶段 |
| 交易类型 | 单独交易 | 单独 + 原子性 bundle |
| 收入 | 基础奖励 + 优先费 | 基础奖励 + 优先费 + MEV 小费 |
| 配置 | 标准 | 额外的区块引擎/Relayer 配置 |

## 10. 争议与批评

### 10.1 私有内存池事件

**核心问题：** Jito 的内存池启用了三明治攻击，损害了零售用户。关闭它将问题推入地下，而非解决问题。

**时间线：**
1. 2022 年 11 月：内存池作为区块引擎的一部分启动
2. 2023 年：Meme 币热潮放大了三明治攻击盈利能力
3. 2024 年 3 月 8 日：创纪录的 10,000 SOL（$150 万）小费，主要由三明治驱动
4. 2024 年 3 月 9 日：Jito 暂停内存池
5. 2024 年 3-6 月：私有内存池出现
6. 2024 年 6 月：Solana 基金会将 32 个三明治验证者列入黑名单
7. 2024 年 6 月：Jito DAO 提出验证者黑名单
8. 2024-2025 年：三明治机器人在 16 个月内提取了 $3.7-5 亿

**不舒服的真相：** Jito 创建了启用 Solana 三明治攻击的基础设施。移除公共内存池并没有停止这种做法 — 只是让它更不透明，利润集中在更少、更精明的运营者手中。

### 10.2 中心化担忧

- **客户端单一文化**：95%+ 的 Solana 质押运行一个修改过的客户端是系统性风险
- **区块引擎作为瓶颈**：所有 MEV 流经 Jito 的链下基础设施
- **200ms relayer 延迟**：创造了结构性信息优势
- **验证者经济**：不运行 Jito 意味着放弃 MEV 收入，创造了经济锁定
- **基金会委托黑名单**：被批评为对"去中心化"网络的中心化治理

### 10.3 三明治攻击辩论

哲学张力：
- **支持 Jito 论点**：MEV 是不可避免的；Jito 至少将其分配给质押者（94% 返还给验证者/质押者），而非让精明的交易者集中获益
- **反对 Jito 论点**：Jito 将 Solana 上的 MEV 提取工业化，创建了大规模启用三明治攻击的基础设施。200ms 延迟实际上就是前跑窗口

### 10.4 验证者激励对齐

- 验证者在经济上被激励运行 Jito-Solana（MEV 小费 = 13-15% 更多奖励）
- 不运行 Jito = 竞争劣势
- 这创造了近乎垄断，减少了客户端多样性（对网络弹性至关重要）
- Firedancer 采用提供了一些缓解，但 Jito + Firedancer 集成（FireBAM）可能延续这一模式

## 12. 诚实评估与可借鉴模式

### 12.1 Jito 真正做得更好的地方

1. **集成 MEV + LST 飞轮**：没有其他协议将 MEV 基础设施与流动性质押结合。飞轮是真实且自我增强的。

2. **与质押者共享收入**：94% 的 MEV 小费返还给验证者和质押者。与以太坊上 MEV 提取通常不成比例地有利于构建者/搜索者相比，Jito 的模型更公平。

3. **开源 MEV 基础设施**：与私有 MEV 运营不同，Jito 发布了其客户端，使提取过程透明可审计。

4. **渐进式去中心化**：TipRouter NCN、JIP-24（100% 收入给 DAO）、StakeNet（链上委托）— 每一步都减少了 Jito Labs 的控制。

5. **BAM 作为工程解决方案**：不依赖政策（内存池关闭），BAM 使用 TEE 使三明治攻击在结构上变得困难。

### 12.2 真正的弱点和风险

1. **客户端单一文化风险**：95%+ 的质押在一个客户端上 = 系统性风险。Jito-Solana 的 bug 可能导致网络停止。

2. **制造了自己要解决的问题**：Jito 的内存池启用了三明治攻击；BAM 是 Jito 构建的基础设施的工程修复。

3. **JTO 代币价值崩溃**：从 ATH 下跌 -94.7%。尽管有巨额收入（$6.74 亿+ 小费），代币持有者并未成比例受益。JIP-24 和 CSD 是修复尝试。

4. **通过经济学实现中心化**：不运行 Jito = 放弃 13-15% 的奖励。这不是去中心化 — 这是经济胁迫。

5. **链下信任假设**：区块引擎和 Relayer 是链下的。验证者信任 Jito 的基础设施是诚实的。BAM 中的 TEE 部分解决了这个问题，但完全验证仍然难以实现。

6. **再质押处于早期**：$2.27 亿 TVL vs EigenLayer 的 $180 亿。NCN 生态系统还在萌芽期 — TipRouter 是唯一的生产级 NCN。

### 12.3 可借鉴的模式

| 模式 | 描述 | 适用性 |
|------|------|--------|
| **MEV + LST 飞轮** | 将 MEV 基础设施与流动性质押结合，使 MEV 收入提高质押收益，吸引更多质押，增加 MEV 覆盖 | 任何有 MEV 的 PoS 链 |
| **部分区块拍卖** | 不使用完整区块构建者拍卖，而是运行粒度化的按 bundle 拍卖，允许每个区块多个获胜搜索者 | PBS 替代方案 |
| **链上委托评分** | 用链上评分程序（StakeNet）替代链下委托机器人，支持无许可 cranking | 任何质押池/委托系统 |
| **渐进式去中心化** | 公司构建 → 开源 → DAO 治理 → 公司放弃收入（JIP-24）→ NCN 去中心化运营（TipRouter） | 协议到 DAO 过渡 |
| **基于 TEE 的交易隐私** | 使用可信执行环境进行交易排序，通过设计（而非政策）防止前跑 | 任何有 MEV 担忧的链 |
| **区块构建插件架构** | 让开发者编写与区块构建者交互的自定义调度逻辑（插件），创造新收入模式 | 区块构建者设计 |
| **Relayer 作为 TPU 代理** | 在用户和区块生产者之间插入代理，提供 MEV 拍卖的短暂延迟，带自动回退 | MEV 协调 |
| **原子 Bundle 原语** | 交易组的全部成功或全部失败执行作为一等原语 | 任何智能合约平台 |
| **NCN 模型** | 通用再质押 → 运营者执行链下工作 → 链上强制执行。比 AVS 更灵活（多资产，SPL 代币支持） | 共享安全系统 |
| **收入到 DAO 管道** | 协议产生收入 → 所有费用流入 DAO 金库 → DAO 管理的子 DAO 部署资本（回购、补贴） | 代币价值累积 |

### 12.4 大局观

Jito 是 Solana 上最重要的第三方基础设施。它控制着 MEV 提取、流动性质押、验证者经济学，以及越来越多的区块构建。这使它既极具价值，又是系统性风险的单一节点。

诚实的张力：**Jito 使 Solana 运行得更好（更高的验证者收入、有竞争力的 MEV 市场、流动性质押），同时将关键基础设施集中在一家公司的栈上。** BAM、TipRouter、StakeNet 和 JIP-24 是朝着去中心化的真诚步骤。但 Jito 飞轮的经济引力使替代方案难以竞争。Firedancer 提供了客户端多样性的最大希望，但 FireBAM 集成意味着即使 Firedancer 验证者也可能运行 Jito MEV 基础设施。

问题不在于 Jito 是好是坏 — 而在于当一个协议生态系统 95%+ 的经济基础设施通过一家公司的软件运行时，无论该软件多么开源和 DAO 治理，这个生态系统是否能健康发展。
